## バージョン管理システムとは

**バージョン管理システム（VCS: Version Control System）**とは、ファイルの変更履歴を記録・管理するシステムです。

### バージョン管理のメリット

```
✅ いつでも過去の状態に戻せる
✅ 誰がいつ何を変更したかわかる
✅ 複数人で同時に作業できる
✅ 変更の経緯を追跡できる
```

### Git が選ばれる理由

```
Git の特徴：
├── 分散型 - 各自が完全な履歴を持つ
├── 高速 - ほとんどの操作がローカルで完結
├── ブランチが軽量 - 気軽に分岐できる
└── 業界標準 - 多くの企業で採用
```

## Git の3つのエリア

Git では、ファイルの状態を3つのエリアで管理します。

### エリアの構造

```
┌─────────────────────────────────────────────────────┐
│                 ワーキングディレクトリ                   │
│              （実際のファイルがある場所）                 │
│                                                       │
│    📄 index.html  📄 style.css  📄 app.js           │
└────────────────────────┬────────────────────────────┘
                         │ git add
                         ▼
┌─────────────────────────────────────────────────────┐
│                  ステージングエリア                     │
│              （コミット準備中のファイル）                │
│                                                       │
│    📄 index.html  📄 style.css                       │
└────────────────────────┬────────────────────────────┘
                         │ git commit
                         ▼
┌─────────────────────────────────────────────────────┐
│                    リポジトリ                          │
│                （コミット履歴の保管庫）                  │
│                                                       │
│    📦 コミット1  📦 コミット2  📦 コミット3            │
└─────────────────────────────────────────────────────┘
```

### 各エリアの説明

| エリア | 英語名 | 説明 |
|:------|:------|:-----|
| ワーキングディレクトリ | Working Directory | 実際に編集するファイルがある場所 |
| ステージングエリア | Staging Area | 次のコミットに含めるファイルを準備する場所 |
| リポジトリ | Repository | コミット（スナップショット）の履歴を保管する場所 |

## コミットとは

### コミット = スナップショット

**コミット**とは、ある時点でのプロジェクト全体の状態を記録したものです。

```
コミットに含まれる情報：
├── 変更されたファイルの内容
├── コミットメッセージ（何を変更したか）
├── 作成者（誰が）
├── 日時（いつ）
└── 親コミットへの参照（どこから派生したか）
```

### コミットの流れ

```javascript
// 1. ファイルを編集
// index.html を修正

// 2. ステージングに追加
$ git add index.html

// 3. コミットを作成
$ git commit -m "ヘッダーのロゴを追加"
```

### コミット履歴のイメージ

```
時間の流れ →

○ ─── ○ ─── ○ ─── ○ ─── ○  (HEAD)
初期   機能A   機能B   修正   最新
コミット  追加    追加
```

各 `○` が1つのコミットを表し、チェーンのようにつながっています。

## ブランチとは

### ブランチ = 作業の分岐

**ブランチ**を使うと、メインの開発ラインに影響を与えずに、別の作業を進められます。

```
                 feature/login
                      ↓
              ○ ─── ○ ─── ○
             /
○ ─── ○ ─── ○ ─── ○ ─── ○ ─── ○  ← main
             \               /
              ○ ─── ○ ─── ○
                      ↑
               feature/signup
```

### ブランチのメリット

```
✅ 機能ごとに独立して開発できる
✅ 本番環境に影響を与えない
✅ 複数人で並行して作業できる
✅ 問題があれば簡単に破棄できる
```

### よく使うブランチ名

```
main（または master）
├── 本番環境用の安定したコード
└── 直接編集しない

feature/○○
├── 新機能の開発用
└── 例: feature/add-login

fix/○○
├── バグ修正用
└── 例: fix/header-layout

docs/○○
├── ドキュメント更新用
└── 例: docs/update-readme
```

## リモートとローカル

### ローカルリポジトリ

```
自分の PC 内にあるリポジトリ
├── 自分だけがアクセスできる
├── オフラインでも作業可能
└── 変更を自由に試せる
```

### リモートリポジトリ

```
サーバー上（GitHub）にあるリポジトリ
├── チームメンバーと共有
├── バックアップになる
└── 本番環境へのデプロイ元
```

### 連携のイメージ

```
┌─────────────────┐         ┌─────────────────┐
│    ローカル       │         │    リモート       │
│  （自分の PC）    │         │   （GitHub）     │
│                   │         │                   │
│  ○ ─── ○ ─── ○  │ ─push→ │  ○ ─── ○ ─── ○  │
│                   │ ←pull─ │                   │
└─────────────────┘         └─────────────────┘
```

| 操作 | 説明 |
|:----|:----|
| push | ローカルの変更をリモートに送る |
| pull | リモートの変更をローカルに取り込む |
| clone | リモートリポジトリをローカルにコピー |

## 図解で理解する Git

### 全体像

```
【開発者 A の PC】          【GitHub】          【開発者 B の PC】
     │                         │                       │
     │                         │                       │
 ローカル                  リモート                 ローカル
リポジトリ                リポジトリ               リポジトリ
     │                         │                       │
     ├──── push ────→          │                       │
     │                         ├───← pull ─────────────┤
     │                         │                       │
     ├───← pull ─────          │                       │
     │                         ├──────── push ────→    │
```

### 1人開発の場合

```
1. ファイルを編集（ワーキングディレクトリ）
2. git add で変更をステージング
3. git commit でコミットを作成
4. git push で GitHub にアップロード
```

### チーム開発の場合

```
1. git pull で最新を取得
2. ブランチを作成
3. ファイルを編集
4. git add → git commit
5. git push でブランチをアップロード
6. Pull Request を作成
7. レビューを受けてマージ
```

## まとめ

Git の基本概念を理解することで、バージョン管理の仕組みがわかります。

### 重要なポイント

```
1. 3つのエリア
   ワーキングディレクトリ → ステージング → リポジトリ

2. コミット = スナップショット
   変更を履歴として記録

3. ブランチ = 作業の分岐
   メインに影響を与えずに開発

4. ローカル ⇄ リモート
   push で送り、pull で取得
```
