## Issue とは

### 定義

**Issue（イシュー）**は、GitHub 上でタスク、バグ報告、機能要望、質問などを管理するための機能です。

```
Issue の用途：
├── タスク管理（やるべきこと）
├── バグ報告（問題の記録）
├── 機能要望（新しいアイデア）
├── 質問・議論
└── ドキュメント作成の依頼
```

### なぜ Issue を使うのか

```
Issue を使うメリット：
├── タスクが可視化される
├── 誰が何をしているかわかる
├── 進捗を追跡できる
├── 議論の履歴が残る
├── PR と紐づけられる
└── 完了条件が明確になる
```

> **Tip**: Slack や Discord だと会話が流れてしまいますが、Issue はいつでも参照できます。

## 良い Issue の書き方

### タイトルのポイント

```
✅ 良い例：
├── 「ログインフォームを作成する」
├── 「ヘッダーのロゴが表示されない不具合」
├── 「検索機能を追加したい」
└── 「README にインストール手順を追記」

❌ 悪い例：
├── 「作業」 ← 何をするかわからない
├── 「バグ」 ← どんなバグかわからない
├── 「機能追加」 ← 具体性がない
└── 「至急」 ← 内容がわからない
```

### 説明のテンプレート（タスク用）

```markdown
## 概要
何をするかの簡単な説明

## 詳細
- 具体的な作業内容1
- 具体的な作業内容2
- 具体的な作業内容3

## 完了条件
- [ ] 条件1を満たしている
- [ ] 条件2を満たしている

## 参考
- 関連するドキュメントや URL
```

### 説明のテンプレート（バグ報告用）

```markdown
## バグの概要
何が起きているか

## 再現手順
1. ○○ページを開く
2. ○○ボタンをクリック
3. ○○と入力する
4. → エラーが発生

## 期待する動作
本来どうなるべきか

## 実際の動作
実際に何が起きたか

## 環境
- OS: macOS 14.0
- ブラウザ: Chrome 120
- Node.js: v20.0.0

## スクリーンショット
（あれば添付）
```

## ラベル（Labels）

### ラベルとは

Issue を分類するためのタグです。色付きで視覚的にわかりやすくなります。

### よく使うラベル

| ラベル | 色 | 用途 |
|:------|:--|:----|
| `bug` | 🔴 赤 | バグ報告 |
| `feature` | 🟢 緑 | 新機能 |
| `enhancement` | 🔵 青 | 改善 |
| `documentation` | 🟣 紫 | ドキュメント |
| `help wanted` | 🟡 黄 | 助けが欲しい |
| `good first issue` | 🟢 緑 | 初心者向け |
| `priority: high` | 🔴 赤 | 優先度高 |
| `priority: low` | ⚪ グレー | 優先度低 |

### カスタムラベルの作成

Settings → Labels から新しいラベルを作成できます。

📸 画像メモ: Labels 設定画面

## 担当者（Assignees）

### 担当者の設定

Issue の右サイドバーから担当者を設定します。

```
担当者を設定するメリット：
├── 誰がやるか明確
├── 自分のタスクをフィルタできる
├── 責任の所在がわかる
└── 進捗確認がしやすい
```

### 複数人での担当

ペアで作業する場合は複数人をアサインできます。

## マイルストーン（Milestone）

### マイルストーンとは

期限を設定して Issue をグループ化できる機能です。

```
例：
├── v1.0.0 リリース（1/31）
├── Week1 までの目標（1/15）
├── MVP デモ準備（2/10）
└── 最終発表（3/1）
```

### 進捗の可視化

マイルストーンを設定すると、完了した Issue の割合がプログレスバーで表示されます。

📸 画像メモ: マイルストーンのプログレスバー

## Issue からブランチを作成

### 手順

Issue ページの右サイドバーにある「Create a branch」をクリックすると、Issue に紐づいたブランチを作成できます。

```
自動生成されるブランチ名の例：
12-add-login-feature
└── Issue番号 + タイトル
```

### メリット

```
✅ Issue との紐づけが自動
✅ ブランチ名を考えなくていい
✅ どの Issue の作業かわかりやすい
```

📸 画像メモ: Create a branch ボタン

## PR で Issue を自動クローズ

### キーワード

PR の説明やコミットメッセージに特定のキーワードを含めると、マージ時に Issue が自動でクローズされます。

```
使えるキーワード：
├── fix #12      / fixes #12
├── close #12    / closes #12
├── resolve #12  / resolves #12
└── 大文字でも OK（Fix #12 など）
```

### 使用例

```markdown
## 概要
ログイン機能を追加しました。

## 関連 Issue
fixes #12

## 変更内容
- LoginForm.jsx を作成
- useAuth フックを追加
```

### 複数の Issue をクローズ

```markdown
fixes #12, fixes #15, fixes #18
```

## 実践：Issue を作成してみよう

### シナリオ

Todo アプリ開発のための Issue を作成します。

### 1. 機能追加の Issue

```markdown
Title: タスクの追加機能を実装する

## 概要
ユーザーが新しいタスクを追加できるようにする

## 詳細
- 入力フォームを作成
- 追加ボタンを実装
- 空の場合はエラーを表示

## 完了条件
- [ ] タスク名を入力できる
- [ ] 追加ボタンでリストに追加される
- [ ] 空欄では追加できない

Labels: feature
Assignees: 自分
```

### 2. ブランチを作成

「Create a branch」から `1-add-task-feature` ブランチを作成

### 3. 開発してPR

```bash
git fetch origin
git checkout 1-add-task-feature

# 開発...

git add .
git commit -m "タスク追加機能を実装 (#1)"
git push origin 1-add-task-feature
```

### 4. PR で自動クローズ

PR の説明に `fixes #1` と記載 → マージで Issue がクローズ

## ミニ演習

### Try 1: Issue を3つ作成する

以下の3つの Issue を作成してみましょう：

1. **機能追加**: 「ダークモードを追加する」
2. **バグ報告**: 「モバイルでボタンがはみ出る」
3. **ドキュメント**: 「README に使い方を追記」

**確認ポイント:**
- [ ] わかりやすいタイトルになっているか
- [ ] 適切なラベルを設定したか
- [ ] 担当者を設定したか

### Try 2: Issue から PR までの流れ

作成した Issue の1つについて、以下を試してみましょう：

1. Issue からブランチを作成
2. 簡単な変更を加えてコミット
3. PR を作成（`fixes #番号` を含める）
4. マージして Issue がクローズされることを確認

## まとめ

Issue を活用することで、タスク管理が効率化できます。

### 活用のポイント

```
Issue の使い方：
├── タスクごとに Issue を作成
├── ラベルで分類
├── 担当者を明確に
├── マイルストーンで期限管理
├── Issue からブランチを作成
└── PR で自動クローズ
```

### ベストプラクティス

```
✅ Issue ファーストで開発（先に Issue を作る）
✅ 1 Issue = 1 タスク（小さく分割）
✅ 完了条件を明記
✅ 関連する Issue は相互リンク
✅ 完了したら必ずクローズ
```
