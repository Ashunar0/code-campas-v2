
## この記事で学ぶこと

- Web アプリ開発の流れ
- 各フェーズの役割
- MVP では最小限でスタート

---

## Web アプリ開発の流れ

```
1. 企画・アイデア出し
   ↓
2. 要件定義（何を作るか）
   ↓
3. 画面設計（どんな見た目か）
   ↓
4. データ設計（どんなデータ構造か）
   ↓
5. タスク分割（どう進めるか）
   ↓
6. 実装
   ↓
7. テスト
   ↓
8. デプロイ・公開
   ↓
9. フィードバック・改善
```

---

## 各フェーズの役割

| フェーズ | 内容 | 成果物 |
|:--------|:----|:------|
| 企画 | アイデアを考える | アプリの概要 |
| 要件定義 | 機能を決める | 要件定義書 |
| 画面設計 | UIを設計 | ワイヤーフレーム |
| データ設計 | DB構造を設計 | データ設計書 |
| タスク分割 | 作業を分解 | Issue一覧 |
| 実装 | コードを書く | 動くアプリ |
| テスト | 動作確認 | バグ修正 |
| デプロイ | 公開 | URL |

---

## 行ったり来たりは OK

実際の開発では、フェーズを行き来することがあります。

```
実装中に仕様の漏れに気づく
  ↓
要件定義に戻る
  ↓
画面設計を修正
  ↓
実装を続ける
```

これは普通のことです。柔軟に対応しましょう。

---

## MVP では最小限でスタート

**MVP（Minimum Viable Product: 実用最小限の製品）** では、すべてを完璧にする必要はありません。

```
優先すること：
├── 動くものを作る
├── 最小限の機能
├── 期限を守る
└── ユーザーに使ってもらう

あとでやること：
├── 細かい機能追加
├── 完璧なデザイン
└── パフォーマンス最適化
```

作りたいアプリを思い浮かべて、以下を考えてみましょう：

1. どのフェーズが最も重要になりそうですか？
2. MVP で削れそうな機能は何ですか？

---

## まとめ

この記事では、Web アプリ開発の全体的な流れを学びました。

- 開発は **企画 → 要件定義 → 設計 → 実装 → テスト → デプロイ** の流れで進む
- 各フェーズには役割があり、成果物がある
- フェーズを行き来するのは普通のこと
- MVP では**最小限の機能**に絞ってスタートする
