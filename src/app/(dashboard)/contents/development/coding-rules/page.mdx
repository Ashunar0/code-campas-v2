## なぜルールが必要か

```
メリット：
├── コードの一貫性が保たれる
├── レビューがしやすい
├── コンフリクトが減る
└── 保守性が上がる
```

---

## チームで決めるべきこと

### 命名規則

```
コンポーネント: PascalCase
  例: TodoItem, UserProfile

ファイル名: PascalCase または kebab-case
  例: TodoItem.jsx, todo-item.jsx

関数: camelCase
  例: handleClick, fetchTodos

定数: UPPER_SNAKE_CASE
  例: MAX_LENGTH, API_URL
```

### ファイル配置

```
src/
├── components/  ← 共通コンポーネント
├── pages/       ← ページコンポーネント
├── hooks/       ← カスタムフック
├── lib/         ← ユーティリティ
└── styles/      ← スタイル
```

### コミットメッセージ

```
形式: [type] 説明

例:
[feature] ログイン機能を追加
[fix] ヘッダーのバグを修正
[docs] README を更新
```
