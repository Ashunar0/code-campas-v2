## GitHub Issues の活用

すべてのタスクを Issue として登録します。

```
メリット：
├── チームで進捗が見える
├── 担当者が明確
├── PR と紐づけられる
└── 完了したら自動クローズ
```

---

## Issue の書き方

### テンプレート

```markdown
## 概要
何をするタスクか簡単に

## 詳細
- やること1
- やること2
- やること3

## 完了条件
- [ ] ○○ができる
- [ ] △△ が表示される
```

### 実例

```markdown
## 概要
ログイン機能を実装する

## 詳細
- Firebase Auth を設定
- ログインフォームを作成
- エラーハンドリングを実装

## 完了条件
- [ ] メールとパスワードでログインできる
- [ ] ログイン後に一覧ページに遷移する
- [ ] エラー時にメッセージが表示される
```

## ラベルの活用

| ラベル | 用途 |
|:------|:----|
| `feature` | 新機能 |
| `bug` | バグ修正 |
| `design` | デザイン |
| `docs` | ドキュメント |
| `priority: high` | 優先度高 |

---

## 担当者を明確に

Assignees で担当者を設定します。

```
ルール：
├── 1つの Issue に 1人の担当
├── 担当者が完了まで責任を持つ
├── 困ったら相談
```

---

## 💡 Try: Issue を書いてみよう

上のテンプレートを使って、あなたのアプリの **1つの機能**の Issue を書いてみましょう。

- 概要、詳細、完了条件の3つを必ず書く
- 完了条件はチェックボックスにする

## まとめ

この記事では、Issue を使ったタスク管理を学びました。

- すべてのタスクを **GitHub Issues** に登録する
- Issue には **概要・詳細・完了条件** を書く
- **ラベル**で分類し、**Assignees**で担当者を明確にする
